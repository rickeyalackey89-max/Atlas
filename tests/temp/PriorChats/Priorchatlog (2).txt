What we were trying to solve in this continuation

* We had already concluded that the old helper backtest (backtest_role_layer_ctx.py) was not a trustworthy canonical backtest for the current model.

How we got here / key discoveries in order

1. The old helper backtest was proven non-canonical

* We compared live scored_legs_deduped.csv against helper backtest outputs and discovered the helper was not representing the same final surface as live.
* Root cause was isolated by tracing:

  * replay snapshot itself was clean,
  * but helper _expand_directions(...) blindly doubled already-directional rows.
* This produced inflated helper report rows and duplicated exact prop keys.
* Fix made:

  * _expand_directions(...) was patched so it only expands rows that are non-directional.
* Important interpretation:

  * that fix made the helper less wrong,
  * but we explicitly decided the helper should remain only a probe/diagnostic tool, NOT the canonical backtest.

2. Architectural decision was locked

* We agreed on the new design:

  * Backtest v2 = faithful replay generator
  * Oracle = reader / tuner / reporter
* Meaning:

  * replay must faithfully reproduce historical today.csv from raw + historical injury state,
  * hard-fail if board construction drifts,
  * then run the CURRENT model on that faithful board,
  * then evaluate against actual outcomes in Last10.csv,
  * then let Oracle read the resulting run folder.
* Very important distinction we locked:

  * replay today.csv must match exactly / behaviorally identically,
  * downstream scored/slip outputs are ALLOWED to differ from historical live outputs if current knobs/config differ,
  * because the point is to test today’s model on historical boards.

3. New Backtest v2 runner was created

* Real implementation target:

  * src/Atlas/model/backtest_v2.py

  * the canvas stub may still show an earlier skeleton,

8. Oracle “possible misses” logic was investigated deeply

* Oracle originally flagged 14 possible misses on MIN.
* User correctly pointed out those were beneficiary legs, not the out player.
* We initially thought:

  * Kyle Anderson had too few games and should be allocator-ineligible.
* Then we checked the real owner:

  * team_share_reallocator.py
* Important discovery from the real owner:

  * min_games_for_pattern = 8
  * min_games_new_team = 10
  * but min_games_for_pattern is not a hard exclusion gate anymore,
  * it is used as a soft reliability scaler:

    * effectively rel = min(1, games_max / min_games_for_pattern)
* Another important correction:

  * share_matrix.csv["games"] is NOT a player-level games-played eligibility source.
  * It is pairwise/teamshare pattern support.
* Therefore:

  * Oracle should NOT infer allocator eligibility from share_matrix.csv["games"] as if it were player GP.
