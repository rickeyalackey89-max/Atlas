How we got here
1) We first finished the Standard baseline work.
- Control, Variant 01, and Variant 02 were compared on replay telemetry.
- Variant 01 won and stayed locked.
- Variant 02 was too aggressive for shorter System slips.

2) We then built a telemetry corpus reader so replay telemetry could judge config + calibration.
- Early reader versions worked, but initially acted more like observers than candidate judges.
- Reader was hardened repeatedly:
  - ranking formula
  - shorter-slip protection
  - per-run stability penalty
  - sample-depth penalty
  - regime/time-window promotion gates
  - support for different corpus layouts
  - support for legacy id shapes

3) We created a telemetry calibration workflow.
- Idea: keep Standard fixed, keep map calibration neutral, and use a late telemetry overlay as the only challenger surface.
- That means:
  - control = no telemetry calibration file
  - challenger = telemetry artifact present
  - same raws, same truth file, same config, same corpus

4) We hit a major wiring bug.
- Early control/challenger corpora came out identical.
- That proved replay was not consuming `telemetry_calibration.json`.
- This was NOT a reader problem.

5) We fixed the telemetry integration seam.
- Telemetry calibration is now wired into the `p_cal` path.
- Replay artifacts now show proof columns:
  - `telemetry_cal_key`
  - `telemetry_k_shrink`
  - `telemetry_under_penalty`
  - `telemetry_mult`
  - `telemetry_bucket_mult`
  - `telemetry_cal_applied`
  - `p_cal_src` appends `+telemetry`
- Single-raw A/B proved telemetry overlays now really fire.

What challengers have already been tested

Why they were rejected
- They either made corpus Brier/log loss worse, or they damaged protected slip surfaces enough that they were not promotable.

What changed the calibration direction
We built and ran a dedicated diagnostic:
- `tools/telemetry_calibration_diagnostic.py`

This was crucial because it changed the interpretation.
The diagnostics showed:
- the main issue is NOT broad overheating
- the biggest recurring errors are underconfident slices, especially:
  - `FG3M UNDER`
  - `RA UNDER`
  - `PRA UNDER`
  - `REB UNDER`
  - `PA UNDER`
  - `PR UNDER`
- also useful underconfidence in:
  - `PR OVER`
  - `PRA OVER`
  - `PTS OVER`
  - `PA OVER`

That means the correct next family is:
- targeted lift
not:
- broad cooling
- broad UNDER penalty
- blunt board-wide moves

Targeted lift family
1) `telemetry_calibration_targeted_lift_v1.json`
- Single-raw A/B: clean pass
- Full corpus A/B: first challenger that actually improved both Brier and log loss on the corpus
- But it still did not auto-promote because the gates remained strict and it did not clear enough slices/windows broadly enough.

2) `telemetry_calibration_targeted_lift_v2.json`
- Built as a slightly softened version of v1.

  - `PRA|OVER = 1.008`
  - `PTS|OVER = 1.006`
  - `PA|OVER = 1.006`
- Single-raw A/B: clean pass
- It is the current best active challenger and the current open question is whether it will promote on the full corpus.
